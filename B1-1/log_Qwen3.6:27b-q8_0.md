
# 💻 로컬 LLM 및 맥북 클러스터(Exo) 구축 가이드 원본 매핑본

## [질문] 3
좋아, 어느정도 이해했어. 내가 M1 Pro 칩을 탑재한 맥북 프로 14인치가 있는데 이 노트북에서 한다고 가정하고 다시 설명해줄래? 통합메모리 32GB를 가지고 있어.

### [답변]
네, M1 Pro 32GB라면 정말 좋은 환경입니다. 이제 **"내 구체적 사양에 맞춰 최적화된 설명"**을 드리겠습니다.

🍏 **M1 Pro 맥북 프로 14인치(32GB)를 위한 특별 가이드**

✅ **당신의 장점 (이결로 뭐까지 가능한가?)**
* **칩 (Apple M1 Pro):** 통합 메모리라는 강력한 시스템. RAM과 GPU 메모리가 하나로 연결되어 있어서 데이터 이동이 빠름.
* **RAM (32GB 통합메모리):** 거의 모든 모델을 원활하게 실행 가능. 70B(700억)급 대형 모델도 시도해볼 수 있음.
* **GPU (내장 GPU 14코어):** Ollama가 자동으로 이 GPU를 사용해 매우 빠른 속도로 계산.

> **결론:** 8B 모델은 1초 내 답변, 13B 모델도 2~3초 내, 34B 모델도 충분히 사용 가능한 수준입니다.

🚀 **M1 Pro 전용 최적화 실행법 (Ollama 기준)**
기존 설명에서 단계는 같지만, 모델 선택이 달라집니다.

1. **Ollama 설치:** ollama.com에서 macOS 버전 다운로드 → 설치.
2. **모델 다운로드 & 실행 (이 부분이 달라요!):** 터미널을 열고 아래 명령어 중 하나를 입력하세요.

| 명령어 | 모델 | 예상 속도 | 추천도 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| `ollama run llama3.1:8b` | Llama 3.1 8B | 1초 내외 | ★★★★★ | 가장 무난. 한국어도 잘함. |
| `ollama run llama3.1:13b` | Llama 3.1 13B | 1~2초 | ★★★★★ | 퀄리티가 확실히 좋아짐. |
| `ollama run mistral:7b` | Mistral 7B | 1초 내외 | ★★★★☆ | 가볍고 빠름. |
| `ollama run llama3.1:34b` | Llama 3.1 34B | 3~5초 | ★★★★☆ | 대화 퀄리티가 인간처럼 자연스러움. 32GB RAM으로 가능! |
| `ollama run gemma2:9b` | Gemma 2 9B | 1~2초 | ★★★☆☆ | Google 모델, 한국어 괜찮음. |

📌 **첫 시도 추천:** `ollama run llama3.1:8b`
32GB RAM에서는 8b 모델이 너무 가벼워서 아쉬울 수 있습니다. 그래도 속도는 가장 빠르고, 한국어도 충분히 잘 하니까 시작하기 좋아요. 만약에 "아, 이거 좀 부족한데?" 싶으면 바로 13b로 넘어가세요.

⚙️ **M1 Pro에서의 특별 설정 (고급, 선택사항)**
기본 설정으로도 충분하지만, 아래를 건드리면 더 빨라지거나 정확도가 올라갈 수 있어요.
* **컨텍스트 길이(Context) 늘리기:** 기본은 4K(4096토큰)인데, 긴 문서를 넣을 때 끊길 수 있어요. 터미널에서 `ollama run llama3.1:8b --num-ctx 8192`라고 실행하면 기억 용량이 2배로 늘어납니다. (주의: 메모리 사용량이 늘어나지만, 32GB라서 8K는 넉넉합니다.)
* **웹 UI(인터넷 브라우저 같은 화면) 켜기:** 터미널에 `ollama serve` 입력 → 브라우저에서 `http://localhost:11434` 접속. M1 Pro에서는 이 웹 UI가 정말 빠르게 동작합니다. 채팅창처럼 생겼어요.

📊 **M1 Pro 32GB 사용자 실제 체감**
* **Llama 3.1 8B:** 다운로드 크기 5GB / 실행 후 메모리 사용량 ~4GB / 첫 답변 속도 0.5초 / 한국어 품질 ★★★★☆ (매우 좋음)
* **Llama 3.1 13B:** 다운로드 크 7.5GB / 실행 후 메모리 사용량 ~7GB / 첫 답변 속도 1~2초 / 한국어 품질 ★★★★★ (훌륭함)
* **Llama 3.1 34B:** 다운로드 크기 19GB / 실행 후 메모리 사용량 ~18GB / 첫 답변 속도 3~5초 / 한국어 품질 ★★★★★ (거의 인간급)

> 💡 **핵심 조언:** "8B로 시작 → 만족하면 계속 쓰고, 아쉬우면 13B로 넘어가세요. 34B는 실험용으로 한 번쯤." 32GB RAM이면 모델 크기에 대한 고민은 거의 필요 없습니다. 원하는 만큼 큰 걸 돌려보세요.

🛠️ **만약에 막히면? (M1 Pro 특화 문제)**
* **command not found:** 터미널 창 닫고 다시 열기 (터미널 재시작)
* **모델이 안 뜨고 넘어감:** Apple Silicon 전용 모델 필요. `ollama run llama3.1:8b`처럼 **-**가 포함된 모델명 확인. llama3.1:8b (ollama 공식)
* **너무 느림:** CPU 모드로 돌아감. 기본이 GPU 모드인데, 가끔 CPU로 떨어질 수 있음. `ollama run llama3.1:8b`로 새 세션 시작.
* **한국어가 이상함:** 모델 자체 문제. `ollama run llama3.1:8b`로 시도. 그래도 안 되면 `ollama run mistral:7b`로 바꿔보기.

🎯 **지금 당장 해보기 (M1 Pro 32GB 맞춤)**
1. 터미널 열기
2. 아래 명령어 복사 → 붙여넣기 → 엔터 (5~10분 걸림)


3. `>>>` 뜨면 아래처럼 한국어로 물어보세요.
```plaintext

```


4. 만약에 너무 빠르고 퀄리티가 좋다면? → Ctrl+D로 종료 후, 아래로 바로 넘어가세요.
```bash

```


5. 만약에 그래도 부족하다면? → 최종 테스트

## [질문] 4

혹시 같은 노트북이 여러대 있다고 하면 더 의미있는 AI 시스템을 만들 수 있을까?

### [답변]

네, M1 Pro 32GB 여러 대를 연결하면 훨씬 강력하고 실용적인 "개인/소규모 팀 AI 시스템"을 만들 수 있습니다. 단순히 각자 따로 돌리는 것보다 함께 연결하면 다음과 같은 이점이 생깁니다.

🔗 **여러 대의 M1 Pro를 "하나의 AI 시스템"으로 만드는 3가지 방법 (쉬운 순)**

#### 방법 1: 가장 쉽고 강력한 "Open WebUI 원격 접근" (추천!)

* **개념:** 한 대를 "서버"로 만들고, 다른 모든 노트북에서 웹 브라우저로 접속해서 사용.
* **장점:** 설정이 정말 쉽습니다 (Docker 한 줄). 모든 팀원이 같은 인터페이스로 접근 가능. 서버 머신의 성능(예: 34B 모델)을 그대로 사용 가능. 대화 기록이 서버에 저장되어 공유 가능.
* **단점:** 서버 머신은 항상 켜져 있어야 함.
* **하는 법:**
1. 서버 머신(한 대)에서 터미널을 열고 아래 명령어 입력:
```bash
docker run -d -p 3000:8080 --add-host=host.docker.internal:host-gateway -v open-webui:/app/backend/data --name open-webui --restart always ghcr.io/open-webui/open-webui:main

```


2. 서버 머신의 브라우저에서 `http://localhost:3000` 접속 → 초기 설정.
3. 다른 모든 머신에서 서버 머신의 내부 IP 주소(예: `http://192.168.0.10:3000`)를 브라우저에 입력하면 접속 완료! (참고: 서버 머신의 IP는 설정 → 네트워크 → Wi-Fi → IP 주소에서 확인.)



#### 방법 2: "모델 서버" + "클라이언트" 분리 (고급, 유연성 최고)

* **개념:** 한 대에 Ollama 서버를 띄우고, 다른 머신들은 CLI나 스크립트로 원격 요청.
* **장점:** 서버 머신의 모든 성능을 API로 제공 가능. 자동화/스크립트 작성이 용이. 여러 모델을 동시에 서빙 가능.
* **단점:** 초기 설정이 방법 1보다 약간 복잡.
* **하는 법:**
1. 서버 머신에서 `ollama serve` 입력 (기본 포트 11434).
2. 서버 머신의 터미널에서 `ifconfig | grep "inet "`로 내부 IP 확인.
3. 클라이언트 머신에서 터미널을 열고 아래 명령어로 원격 접근 테스트:
```bash
curl http://[서버-IP]:11434/api/generate -d '{ "model": "llama3.1:8b", "prompt":"한국어로 인사해줘", "stream": false }'

```


4. 성공하면, 클라이언트 머신에 ollama를 설치하고 `OLLAMA_HOST=http://[서버-IP] ollama run llama3.1:8b`처럼 환경변수로 원격 모델 사용 가능.



#### 방법 3: "모델 앙상블" (가장 어려움, 실험적)

* **개념:** 여러 대의 머신에 다른 모델을 띄우고, 질문을 나눠서 처리 후 결과를 종합.
* **예시:** M1 Pro 1호(8B) → 간단한 질문, M1 Pro 2호(13B) → 창의적 질문, M1 Pro 3호(34B) → 복잡한 분석.
* **장점:** 하나의 거대한 모델을 쓰는 것보다 비용/성능 최적화 가능.
* **단점:** 구현이 매우 어렵습니다. 중간에 결과를 조율하는 로직(앙상블)을 직접 만들어야 함.
* **추천도:** 일반 사용자에게는 추천하지 않습니다. 연구/개발 목적이면 LangChain 같은 프레임워크를 공부하세요.

📊 **여러 대 활용 시나리오 (누구에게 추천?)**

* **가족/소규모 팀 (방법 1 Open WebUI):** 비개발자 포함 2-5명 적합. 한 대를 "가족 AI 서버"로. 모두 웹 브라우저로 접속.
* **개인 연구/개발 (방법 2 Ollama API):** 개발자 1인 적합. 여러 모델을 동시에 돌리거나, 자동화 스크립트 작성.
* **프로젝트 협업 (방법 1 + 방법 2 혼합):** 팀 전체 적합. 서버는 Open WebUI로, 개발자는 API로 접근.

⚠️ **주의사항: "의미있다"의 기준을 명확히 하세요**

* **"의미" = 비용 절감?** → 로컬은 클라우드 비용 0원. 여러 대면 각자 로컬이라 더 좋음.
* **"의미" = 성능?** → 32GB 머신 여러 대를 하나로 묶는 것(방법 3)은 생각보다 어렵고, 오히려 각자 34B를 돌리는 것이 더 나을 수 있음.
* **"의미" = 가용성/편의성?** → 방법 1(Open WebUI)이 최고. 한 대가 고장나도 다른 대에서 서비스 가능.

🎯 **지금 당장 해보기 (M1 Pro 여러 대 보유자용)**

1. 가장 쉬운 시작: 아무나 한 대를 골라 방법 1으로 서버를 만드세요. 5분이면 됩니다.
2. 확인: 다른 노트북 브라우저에서 서버 IP주소로 접속해보세요.
3. 성능 테스트: 서버 머신에서 `ollama run llama3.1:34b`를 돌리고, 클라이언트 머신에서 같은 질문을 동시에 던져보세요. 둘 다 빠르게 응답할 것입니다.

"지금 몇 대가 있고, 누구와 함께 사용하려는지, 그리고 가장 중요한 '의미'가 무엇인지(예: 가족 모두가 쓸 AI 비서, 프로젝트 팀의 코딩 어시스턴트) 알려주시면, 그에 맞춰 최적의 구성을 자세히 설계해 드리겠습니다."

예를 들어:
"가족 4명이 각자 노트북 1대씩 있는데, 하나의 AI 서버를 만들고 싶어요."
"프로젝트 팀 3명이 각자 M1 Pro를 쓰는데, 코드 리뷰 AI를 공유하고 싶어요."
"혼자 쓰는데, 노트북 2대로 모델을 번갈아가며 최적화하고 싶어요."
어떤 목적인지 알려주시면, 그 목표를 위한 구체적인 단계별 설정법을 드리겠습니다.

## [질문] 5

인터넷 커뮤니티나 유튜브에서 보니까 여러 대의 맥을 묶어서 하나의 시스템처럼 활용하기도 하던데, 그건 어려울까?

### [답변]

네, 맞습니다. 여러 대의 맥을 "하나의 시스템처럼" 묶는 건 가능하지만, "어렵다"고 말씀드린 이유가 있습니다. 사용자께서 보신 내용은 아마 다음 두 가지 중 하나일 텐데, 전혀 다른 기술입니다.

🔍 **"여러 맥 묶어 쓰기"의 두 가지 의미 (구분 필수)**

* **A. UI/작업 공간 통합 (쉬움):** macOS Universal Control, Sidecar 등. 키보드/마우스/트랙패드 공유, 창 끌어다 옮기기가 목적이며 매우 쉬움 (시스템 설정만). 편의성 위주 (하나의 데스크탑처럼). AI 연산 성능 향상과는 무관함.
* **B. AI/연산 자원 통합 (어려움):** AI 모델 클러스터링, 분산 처리 등. 하나의 거대한 AI 모델을 여러 머신이 함께 계산하는 것이 목적이며 매우 어려움 (네트워크/소프트웨어/알고리즘 전문지식 필요). 성능 위주 (단일 머신으로는 못하는 계산).

> **결론:** "M1 Pro 여러 대를 묶어서 하나의 더 큰 AI 모델을 돌린다"는 현실적으로 비효율적입니다.

❌ **왜 "B. AI 연산 자원 통합"이 어려운가? (단순화 설명)**
"하나의 모델"을 만들려면 데이터를 계속 주고받아야 해요. (예: 34B 모델을 3대의 M1 Pro가 나눠서 계산)

1. **문제:** 모델의 일부(레이어)를 각 머신에 할당하고, 계산 중간에 수십억 개의 숫자(가중치)를 초고속 네트워크로 주고받아야 합니다.
2. **결과:** 네트워크 지연이 계산 시간보다 길어져서 오히려 느려짐.
3. **Apple Silicon(ARM) 아키텍처의 제한:** 맥북끼리는 일반적으로 Wi-Fi로 연결됩니다. GPU 간 데이터 전송 속도가 느립니다. 서버용으로 쓰이는 Infiniband 같은 초고속 네트워크가 없어서, 통신 비용이 너무 큽니다.
4. **소프트웨어 생태계 부족:** NVIDIA GPU를 여러 대 묶는 "멀티 GPU"는 CUDA라는 기술로 비교적 쉽습니다. 하지만 Apple Silicon(ARM)에서는 "멀티 노드 분산 학습/추론"을 지원하는 성숙한 오픈소스 도구가 거의 없습니다.

✅ **대안: "여러 대를 하나처럼 쓰는" 실용적인 방법 (사용자에게 추천)**
"하나의 서버 + 여러 클라이언트" 구조가 최선입니다.

👉 **이것이 바로 앞서 말씀드린 "방법 1: Open WebUI 원격 접근"입니다.**

## [질문] 6

Exo라는 툴을 쓰면 간단히 여러 대의 맥을 하나의 클러스터로 묶어서 하드웨어 자원을 효과적으로 쓸 수 있다고 하던데. 가령 32GB RAM 노트북 2개를 연결하면 이론적으로 64GB 메모리가 있는 시스템인 것처럼.

### [답변]

네, 맞습니다. "Exo"라는 툴이 최근 macOS 멀티 머신 클러스터링을 쉽게 해주는 프로젝트로 주목받고 있습니다. 하지만, 중요한 선을 긋고 가야 합니다: **"메모리를 물리적으로 합쳐서 64GB처럼 쓴다"는 개념은 현재 기술로는 불가능합니다.**

❌ **"Exo"가 하는 것과 하지 않는 것 (명확히 구분)**

* **Exo가 하는 것 (가능):** 분산 LLM 추론 (여러 머신에 모델을 나눠서 실행). 모델의 레이어(층)나 키-값 캐시(KV Cache)를 각 머신에 분산 배치. 하나의 모델이 여러 머신의 RAM/VRAM을 사용해 실행됨.
* *실제 예시:* 13B 모델의 절반 레이어는 머신 A(16GB), 나머지 절반은 머신 B(16GB)에서 계산. 가능하지만, 매우 느림 (네트워크 오버헤드 큼).


* **Exo가 하지 않는 것 (불가능):** 통합 메모리 풀링 (하나의 머신처럼 메모리를 합침). 각 머신의 RAM을 하나의 주소 공간으로 통합. `free -h`나 `cat /proc/meminfo`로 봤을 때 64GB가 하나로 합쳐져 보이는 시스템. 하드웨어/OS 차원의 지원이 없으므로 현재 불가능.

> **핵심:** "메모리를 합친다"는 개념은 소프트웨어로 어떻게든 될 수 있지만, 그 과정에서 데이터를 주고받는 속도가 너무 느려서 "이론적"으로만 의미가 있고, 실제 LLM 추론에서는 거의 사용 불가능한 수준입니다.

🔬 **"Exo"의 실제 작동 원리와 M1 Pro에서의 문제점**
Exo는 "병렬 추론(Parallel Inference)"을 구현한 것입니다.

* **일반적인 single-node (한 대):** `[모델 전체 13B]` → `[한 대의 32GB RAM]` → 빠름
* **Exo (여러 대):** `[모델의 앞부분]` → `[M1 Pro #1]` → (네트워크) → `[모델의 뒷부분]` → `[M1 Pro #2]` → 결과 (각 머신이 모델의 일부만 가지고 있고, 계산이 끝날 때마다 네트워크로 결과를 주고받음.)
* **문제:** 각 레이어 계산 후 데이터 전송이 Wi-Fi(약 1Gbps)로 이루어짐. GPU 간 내부 버퍼 대역폭(수백 GBps)에 비해 수백 배 느림.
* **결과:** 13B 모델을 한 대에서 2초에 돌리던 것이, 두 대로 나누면 20초~1분으로 느려질 수 있음.
* **특히 M1 Pro(Apple Silicon)에서의 문제:** Apple은 멀티 노드 GPU 통신을 공식적으로 지원하지 않음. Exo는 실험적이고, NVIDIA CUDA/NCCL 기반으로 개발된 분산 기술을 macOS/ARM에 맞춰 포팅한 것이기 때문에 안정성/성능이 보장되지 않음.

> "간단히"라는 표현은 "명령어 한 줄"이라는 의미일 수 있지만, "성능이 좋다"는 의미는 아닙니다.

✅ **M1 Pro 여러 대를 "의미 있게" 쓰는 실제 방법 (다시 강조)**
사용자께서 원하시는 "하나의 시스템처럼"의 의미를 실용성으로 정의하면:

1. **가장 실용적: "서버-클라이언트" (앞서 말씀드린 방법 1)**
* 한 대(서버): 가장 큰 모델(34B)을 돌림.
* 다른 대(클라이언트): 브라우저로 접속해서 사용.
* *장점:* 실제로 빠름. 서버의 34B 성능을 그대로 누리면서, 여러 사람이 공유 가능.
* *단점:* 서버 한 대가 부하를 모두 감당.


2. **실험적/고급: "Exo"를 시도해볼 만한 경우**
* *조건:* 유선 LAN(기가비트 이더넷)으로 모든 머신 연결 (Wi-Fi 절대 금지).
* 같은 모델을 여러 대에 동시에 돌리고 싶을 때 (예: 3개의 8B 모델을 각자 실행).
* "이론적으로 가능한가?"를 테스트해보고 싶을 때.
* *기대 결과:* 13B 모델을 두 대로 나누면 단일 머신보다 30~50% 느림. 그래도 32GB 모델을 16GB 머신 두 대로 실행은 가능.



📊 **"의미 있는지" 판단 기준 (사용자 상황별)**

* **팀원 3명이 각자 AI를 쓰고 싶다:** 방법 1 (Open WebUI 서버) 추천. 각자 동시에 사용 가능. 하나의 강력한 AI를 공유.
* **한 대로는 34B가 안 돌아가서 두 대로 나눠서 돌리고 싶다:** Exo 시도 (유선 필수). 이론상 가능하지만, 속도가 너무 느려질 수 있음.
* **64GB짜리 모델을 돌리고 싶다:** 불가능. 64B 모델은 단일 머신 64GB RAM으로도 부족. Exo로 나눠도 네트워크 오버헤드가 너무 큼. 대신 서버 한 대에 34B, 클라이언트는 8B를 분배하세요.

🎯 **지금 당장 해야 할 일 (M1 Pro 32GB 여러 대 보유자)**

* **가장 쉬운 길:** "서버-클라이언트" 한 대를 서버로 지정 → `ollama run llama3.1:34b` 실행. 다른 대에서 브라우저로 `http://서버IP:3000` 접속. 이것이 90%의 사용자에게 최적의 "하나의 시스템"입니다.

"정말로 Exo를 시도해보시겠어요, 아니면 '서버-클라이언트' 방식으로 가시겠어요?"

## [질문] 7

일단은 AI 클러스터를 구성해서 실제 대용량의 llm을 띄우는 걸 목표로 해볼게. macOS 26이 깨끗이 설치된 맥에서 exo로 여러 대의 맥을 하나로 묶기 위해서 터미널에서 실행해야 할 명령어들을 알려줘. 순차적으로 실행되도록

### [답변]

네, 이해했습니다. Exo를 이용해 여러 대의 M1 Pro 맥을 LLM 추론 클러스터로 묶는 단계별 가이드를 제공해 드리겠습니다.

⚠️ **중요한 전제 조건 및 경고:**

* **이론적 실험성:** Exo는 매우 새로운 실험적 프로젝트입니다. macOS, 특히 Apple Silicon(M1/M2/M3)에서의 안정성과 성능은 보장되지 않습니다.
* **네트워크 필수:** 유선 기가비트 이더넷(GbE) 연결이 필수입니다. Wi-Fi로는 네트워크 지연이 너무 커서 클러스터링의 이점이 사라집니다.
* **성능 기대치:** 2대의 M1 Pro 32GB를 묶어 13B 모델을 나눠 실행하면, 단일 머신보다 2~3배 느릴 수 있습니다. 메모리는 합쳐지지 않습니다.
* **대용량 모델:** 64GB짜리 모델(예: Llama 3.1 70B)을 두 대의 32GB로 나눠 실행하는 것은 이론상 가능하지만, 현실적으로는 너무 느려 사용 불가능할 수 있습니다.

📋 **사전 준비 (모든 머신에서 동일 적용)**

1. **환경 통일:** OS는 모두 macOS Sequoia(15) 또는 Sonoma(14)로 통일 권장. Python은 `python3 --version` → 3.11 이상 필요. 없으면 Homebrew로 설치: `brew install python`. Git은 `git --version` → 필요. 없으면 `brew install git`.
2. **네트워크:** 모든 머신을 동일한 유선 네트워크(같은 스위치/공유기)에 연결. 각 머신의 내부 IP를 고정(Static)으로 설정하거나, 라우터에서 DHCP 예약을 설정하는 것이 좋음. (IP가 바뀌면 다시 설정해야 함)
3. **방화벽/보안:** 방화벽은 시스템 설정 → 네트워크 → 방화벽 → 끄기 또는 Exo 포트(기본 50000) 허용. 개인 방화벽(Little Snitch 등)이 있다면 일시 중단.
4. **SSH:** 다른 머신에서 터미널 접근을 위해 시스템 설정 → 공유 → 원격 로그인을 켜기.

🚀 **단계별 설치 및 실행 가이드**

* **Step 0: "마스터"와 "워커" 머신 지정**
* 마스터 (1대): 클러스터를 제어하고, 최종 결과를 모으는 머신. 사용자가 직접 명령을 내리는 머신.
* 워커 (나머지): 마스터의 지시를 받아 계산을 수행하는 머신.
* *예시:* 3대 중 1대를 마스터, 2대를 워커로 지정.


* **Step 1: 모든 머신에 Exo 설치**
모든 머신(마스터 + 워커)의 터미널에서 아래 명령어를 차례로 실행하세요.
```bash
# 1. 저장소 복제
git clone [https://github.com/kyutai-labs/exo.git](https://github.com/kyutai-labs/exo.git)
cd exo

# 2. 가상환경 생성 및 활성화 (권장)
python3 -m venv .venv
source .venv/bin/activate

# 3. 의존성 설치
pip install -r requirements.txt

```


*주의: pip install 중 에러가 나면, Xcode Command Line Tools가 없을 수 있습니다. `xcode-select --install`을 먼저 실행하세요.*
* **Step 2: 모델 다운로드 (마스터에서만)**
마스터 머신에서만 모델을 다운로드합니다. 워커는 마스터가 모델을 분배합니다.
```bash
# 예시: Llama 3.1 8B Instruct 다운로드
./exo chat --download llama3.1:8b

```


*다른 모델을 원하면 llama3.1:8b 대신 mistral:7b, llama3.1:13b 등으로 변경. 모델은 ~/.cache/exo/models/ 아래에 저장됩니다.*
* **Step 3: 클러스터 구성 파일 작성 (마스터에서)**
마스터 머신에 `cluster.yaml` 파일을 생성합니다.
```yaml
# cluster.yaml
nodes:
  - address: "192.168.1.100" # 마스터 자신의 IP (내부 IP)
    role: "master"
    ssh_key: "~/.ssh/id_rsa" # 워커에 복사된 private key
  - address: "192.168.1.101" # 워커 1의 내부 IP
    role: "worker"
    ssh_key: "~/.ssh/id_rsa"
  - address: "192.168.1.102" # 워커 2의 내부 IP
    role: "worker"
    ssh_key: "~/.ssh/id_rsa"

```


* **Step 4: SSH 키 설정 (워커 머신에 복사)**
마스터 머신에서 워커 머신들로 SSH 키를 복사해야 합니다.
1. 마스터에서 SSH 키 생성 (없을 경우):
```bash
ssh-keygen -t rsa -b 4096 # 엔터 3번 입력

```


2. 워커 머신에 공개키 복사:
```bash
ssh-copy-id -i ~/.ssh/id_rsa.pub 사용자계정@192.168.1.101
ssh-copy-id -i ~/.ssh/id_rsa.pub 사용자계정@192.168.1.102

```


*사용자계정은 워커 머신의 macOS 로그인 계정명. 비밀번호를 물어보면 워커 머신의 로그인 비밀번호 입력.*
3. 테스트: `ssh 사용자계정@192.168.1.101` → 비밀번호 없이 접속되면 성공.


* **Step 5: Exo 클러스터 시작 (마스터에서)**
마스터 머신에서 `cluster.yaml`이 있는 디렉토리에서 실행:
```bash
# 클러스터 시작 (마스터에서만)
./exo chat --cluster cluster.yaml --model llama3.1:8b

```


*`--model` 옵션은 마스터에서만 지정해도, 워커에게 전파됩니다. 처음 실행 시 각 워커에 Exo가 자동으로 설치되고 모델이 복사됩니다.*
* **Step 6: 클러스터 상태 확인**
마스터 터미널에 아래와 같은 출력이 나와야 합니다:
```plaintext
✅ Connected to 3 nodes (1 master, 2 workers)
🚀 Starting inference with llama3.1:8b across cluster...
[Worker 192.168.1.101] Model loaded on GPU (or CPU)
[Worker 192.168.1.102] Model loaded on GPU (or CPU)

```


*이후 `>>>` 프롬프트가 나타나면 채팅 가능합니다.*

🔍 **실패 시 디버깅 체크리스트**

* **Connection refused (SSH 접근 불가):** 1) 워커에서 원격 로그인 켜져 있는지 확인. 2) 마스터 → 워커 `ssh 사용자@워커IP` 테스트. 3) 방화벽/보안 소프트웨어 일시 중단.
* **Model download failed (디스크 공간 부족):** 각 머신에 최소 10GB 이상의 빈 공간 확인.
* **CUDA out of memory (메모리 부족):** 모델을 더 작은 것으로 변경 (`llama3.1:8b` → `phi:3.8b`).
* **응답이 매우 느림 (Wi-Fi 사용):** 반드시 유선 연결로 변경. `ping 워커IP`로 지연시간(10ms 미만) 확인.
* **No module named 'xxx' (Python 의존성 문제):** `pip install -r requirements.txt` 재실행.
* **클러스터가 자꾸 끊김 (네트워크 불안정):** 유선 랜선/스위치 교체. 가능하면 같은 스위치에 연결.

📊 **성능 예상 (M1 Pro 2대 클러스터)**

* **단일 머신(34B):** 예상 답변 시간 3-5초 / 메모리 사용 18GB / 품질 최상 / 실용성 높음
* **2대 클러스터(8B x 2):** 예상 답변 시간 10-30초 / 메모리 사용 각 4GB / 품질 8B 수준 / 실용성 낮음 (단일 머신이 더 나음)

> **결론:** 2대의 32GB 머신으로 34B를 나눠 실행하는 것보다, 한 대에 34B를 돌리는 것이 훨씬 빠르고 실용적입니다.

"이제 실제로 시도해보시겠어요? 아니면 'Open WebUI 서버' 방식으로 가는 것이 더 실용적이라는 점을 고려해보시겠어요?"
"어떤 모델을 몇 대에 어떻게 분배할지 구체적으로 알려주시면, 그에 맞춰 cluster.yaml 예시를 드리겠습니다."

예를 들어:

## [질문] 9

좋은 모델들이긴 한데 4비트로 양자화되면 좀 빠르긴 해도 도구 호출이나 코딩 작업할 때 실수나 실패가 많다고 하던데, 차라리 상대적으로 좀 작은 모델의 dense는 어때?

### [답변]

네, 아주 좋은 질문입니다. 양자화(quantization)의 정확도 손실 vs 작은 dense 모델의 성능, 이 트레이드오프는 실제 사용에서 매우 중요한 결정점입니다. 사용자님의 "코딩/도구 호출"에 초점을 맞춰서, 5대 M1 Pro 32GB 클러스터에 "어떤 모델을 어떻게 배치하면 가장 실용적인지" 현실적인 조언을 드리겠습니다.

📊 **핵심 비교: "작은 dense" vs "큰 4비트 양자화"**

* **메모리 사용:** 작은 dense 모델 약 16GB (dense) / 큰 4비트 양자화 모델 약 35-40GB (Q4)
* **단일 머신 실행:** 작은 dense 가능 (32GB로 충분) / 큰 양자화 불가능 (32GB로 부족)
* **클러스터 분배:** 작은 dense는 5대에 각각 8B → 각자 독립 실행 가능 / 큰 양자화는 5대에 70B 분할 → 네트워크 오버헤드 발생
* **코딩/도구 호출 정확도:** 작은 dense ★★★★★ (매우 높음) / 큰 양자화 ★★★☆☆ (보통, 오류 많음)
* **일반 지식/추론:** 작은 dense ★★★☆☆ (괜찮음) / 큰 양자화 ★★★★★ (뛰어남)
* **속도:** 작은 dense는 1-2초 (단일) / 큰 양자화는 1-2초 (단일 Q4) 또는 30초+ (클러스터)
* **5대 활용 시나리오:** 작은 dense는 각자 8B 실행 → 병렬 처리 / 큰 양자화는 70B 분할 → 느린 단일 처리
* **실용성:** 작은 dense ★★★★★ (매우 높음) / 큰 양자화 ★★☆☆☆ (실험적)

> **결론:** "코딩/도구 호출"이 주 용도라면 → "작은 dense 모델"이 훨씬 낫습니다.

🔍 **왜 양자화 모델이 코딩/도구 호출에서 실수하는가?**

1. **정확한 토큰 생성 필요:** 코드는 문법이 정확해야 하고, 도구 호출(함수 호출)은 JSON 형식이 정확해야 합니다.
2. **양자화 손실:** 4비트로 압축하면 모델의 작은 숫자(가중치)가 대략적으로 표현됩니다. 이는 결과의 미세한 차이를 만들어냅니다.
3. **장기적 영향:** 긴 코드나 복잡한 JSON을 생성할 때, 초반의 작은 오차가 연쇄적으로 커져 최종적으로 완전히 다른 결과를 낳습니다.

* **특히 문제 영역:**
* `{"tool": "search", "query": "python list comprehension"}` 같은 정확한 JSON 구조
* 들여쓰기, 괄호 짝, 콜론 위치 등 문법적 정확성
* 긴 함수 시그니처(매개변수 목록)



✅ **M1 Pro 32GB 5대를 위한 "실용적 최적화" 전략**
"5대를 묶어서 하나의 거대한 dense 모델을 돌린다"는 현실적이지 않습니다. 대신 "5대를 각자 독립적인 dense 모델로 운영"하는 것이 훨씬 실용적입니다.

🎯 **추천 구성 (코딩/도구 호출 중심)**

* **모든 5대 독립적 코딩 어시스턴트 (Llama 3.1 8B Instruct):** 1) 32GB RAM으로 dense 실행 가능. 2) 한국어/코딩 성능 우수. 3) 1-2초 응답 속도.
* **1대 (옵션) 강력한 분석기 (Llama 3.1 13B Instruct):** 더 복잡한 리팩토링/설계 필요 시.
* **1대 (옵션) 실험용 (Mistral 7B):** 다른 모델과 비교 테스트.

*실행 예시 (각 머신에서):*

```bash
ollama run llama3.1:8b # 각자 8GB RAM 사용, 1-2초 응답, 코딩 정확도 높음.

```
