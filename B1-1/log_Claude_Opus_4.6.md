## Q4. 여러 대의 노트북으로 더 의미있는 AI 시스템 구성 가능 여부

**사용자:**
> 혹시 같은 노트북이 여러대 있다고 하면 하나보다는 더 의미있는 AI 시스템을 만들 수 있을까?

**Claude Opus:**

재미있는 질문이에요! 😄

**짧은 답변: 가능은 하지만, 개인 사용자에게는 비추천**

### 가능한 이유

여러 대의 컴퓨터를 네트워크로 연결해서 하나의 큰 AI처럼 쓰는 기술이 있어요.

예를 들어 **Exo**라는 도구를 쓰면
- 맥북 2대의 메모리(32GB + 32GB = 64GB)를 합쳐서
- 혼자서는 못 돌리는 더 큰 모델을 실행할 수 있어요

### 비추천하는 이유

| 문제 | 설명 |
|------|------|
| 속도 저하 | 컴퓨터끼리 데이터를 주고받는 시간이 생겨서 오히려 느려질 수 있어요 |
| 설정 복잡 | 네트워크, 소프트웨어 설정이 꽤 까다로워요 |
| 효율 낮음 | 노트북 2대 값이면 훨씬 좋은 데스크톱 1대를 살 수 있어요 |

> 💡 **현실적인 추천**
>
> 맥북 1대로도 충분히 쓸만한 AI를 돌릴 수 있어요! 32GB면 꽤 똑똑한 모델을 돌릴 수 있거든요.
> 나중에 더 큰 모델이 필요해지면, 그때 GPU가 좋은 데스크톱 1대를 마련하는 게 훨씬 효율적이에요.

그러면 이제 본론으로 돌아가서, 아까 여쭤본 2가지 질문에 답해주시면 바로 설치 가이드 드릴게요! 😊
- 주로 뭐에 쓰고 싶은지
- 한국어 대화가 중요한지

## Q5. macOS 26 + Exo 클러스터 구성 명령어

**사용자:**
> 일단은 AI 클러스터를 구성해서 실제 대용량의 llm을 띄우는 걸 목표로 해볼게. macOS 26이 깨끗이 설치된 맥에서 exo로 여러 대의 맥을 하나로 묶기 위해서 터미널에서 실행해야 할 명령어들을 알려줘. 순차적으로 실행되도록

**Claude Opus:**

도전적인 목표네요! 좋습니다 💪

> ⚠️ **먼저 알아두실 점**
>
> macOS 26은 아직 출시되지 않은 버전이에요.
> - 현재 최신은 **macOS 15 (Sequoia)** 입니다
> - 혹시 macOS 15 또는 macOS 16 (2025년 가을 예정)을 말씀하신 건가요?
>
> 또한 Exo는 아직 활발히 개발 중인 프로젝트라서, OS 버전에 따라 설치 방법이 달라질 수 있어요.

### 모든 맥북에서 동일하게 실행 (macOS 15 Sequoia 기준)

**Step 1: Homebrew 설치**
> Homebrew는 맥에서 프로그램을 쉽게 설치하게 해주는 도구예요

```

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

```

> 설치 끝나면 터미널에 나오는 안내 메시지대로 PATH 설정도 해주세요.

**Step 2: Python 설치**
> Exo는 Python으로 만들어져서 Python이 필요해요

```

brew install python@3.12

```

```

# 설치 확인

python3 --version

# Python 3.12.x 같은 숫자가 나오면 성공!

```

**Step 3: Git 설치**
> Git은 프로그램 소스코드를 다운받는 도구예요

```

brew install git

```

**Step 4: Exo 소스코드 다운로드**

```

cd ~

git clone https://github.com/exo-explore/exo.git

cd exo

```

**Step 5: 가상환경 만들기 & 접속**
> 가상환경은 Exo 전용 작업 공간을 따로 만드는 거예요. 다른 프로그램이랑 충돌 안 나게 해줍니다.

```

python3 -m venv .venv

source .venv/bin/activate

# 터미널 앞에 (.venv) 가 붙으면 성공!

```

**Step 6: Exo 설치**

```

pip install -e .

# 시간이 좀 걸릴 수 있어요. 에러 없이 끝나면 OK!

```

### 클러스터 실행

```

# 맥북 1번 (메인)에서:

exo run

# 맥북 2번, 3번... (나머지)에서도 동일하게:

exo run

```

> 같은 명령어를 모든 맥에서 실행하면 됩니다!
> Exo가 같은 Wi-Fi 안에 있는 다른 맥을 자동으로 찾아서 연결해요.

### 연결 확인

메인 맥북 터미널에 이런 메시지가 보이면 성공:

```

Found peer: [맥북2 이름] (...)

```

### 모델 실행 예시

```

# 클러스터가 연결된 후, 메인 맥북에서:

exo run llama-3.1-8b

# 32GB × 2대 = 64GB면 70B(700억) 파라미터급 모델도 시도 가능:

exo run llama-3.1-70b

```

### 주의사항

| 항목 | 설명 |
|------|------|
| Wi-Fi | 모든 맥이 같은 네트워크에 있어야 해요 |
| 방화벽 | 시스템 설정 → 네트워크 → 방화벽이 꺼져있는지 확인 |
| 첫 실행 | 모델 다운로드 때문에 시간이 오래 걸려요 (수십 GB) |
| 발열 | 노트북이 뜨거워질 수 있어요. 통풍 잘 되는 곳에서! |

> 혹시 실행하다가 에러 메시지가 나오면 그대로 복사해서 보여주세요. 같이 해결해볼게요! 😊

## Q7. 4비트 양자화 대신 작은 Dense 모델 추천

**사용자:**
> 좋은 모델들이긴 한데 4비트로 양자화되면 좀 빠르긴 해도 도구 호출이나 코딩 작업할 때 실수나 실패가 많다고 하던데, 차라리 상대적으로 좀 작은 모델의 dense는 어때?

**Claude Opus:**

아주 좋은 판단이에요! 👏

양자화의 단점을 정확히 짚으셨어요.

### 왜 이 판단이 맞는지

```

큰 모델 + 4bit 양자화  vs  작은 모델 + 원본(dense)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

70B 4bit                    32B dense(원본)

├─ 일상 대화: 좋음           ├─ 일상 대화: 좋음

├─ 도구 호출: ❌ 실수 잦음    ├─ 도구 호출: ✅ 안정적

├─ 코딩: ❌ 미묘한 버그       ├─ 코딩: ✅ 정확함

└─ JSON 출력: ❌ 형식 깨짐    └─ JSON 출력: ✅ 깔끔

```

> **양자화란?** 모델의 숫자 정밀도를 낮춰서 크기를 줄이는 거예요.
> 4bit는 원본(16bit) 대비 1/4 크기지만, 섬세한 작업에서 정확도가 떨어져요.

```

맥1 ←──Wi-Fi──→ 맥2 ←──Wi-Fi──→ 맥3 ...

```

5대를 Wi-Fi로만 연결하면 모델이 메모리에 올라가도 맥끼리 데이터를 주고받는 속도가 발목을 잡을 수 있어요.

### 해결법: Thunderbolt 직접 연결

예산이 허락한다면 Thunderbolt 케이블로 맥끼리 직접 연결하면 Wi-Fi 대비 수십 배 빠른 전송 속도를 얻을 수 있어요.

| 연결 방식 | 대략적 속도 |
|----------|------------|
| Wi-Fi 6 | ~1 Gbps |
| Thunderbolt 3 | ~40 Gbps |

> 이건 나중에 속도가 아쉬울 때 고려해보시면 돼요.

그 외에는 직접 해보시면서 궁금한 게 생길 때 물어보시는 게 가장 효율적이에요. 실행하다가 에러 메시지나 예상과 다른 동작이 나오면 언제든 편하게 가져오세요!
